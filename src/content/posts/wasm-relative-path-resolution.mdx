The `@wasm_module` decorator lets you load a compiled WebAssembly module into a Guppy
program. You give it a path to a `.wasm` file:

```python
@guppy.wasm_module("arith.wasm")
class Arith:
    def add(x: int, y: int) -> int: ...
```

For most projects this works fine. But there's a class of setup where it silently breaks:
any time the file using `@wasm_module` is imported from a different directory than where
it lives. The most common case is a Jupyter notebook sitting one level above the module
that defines the WASM integration.

PR #1496 traces the call stack to find where the user's file actually is, and resolves
paths from there.

## The bug: paths resolved from CWD

When Python resolves a relative path, it resolves it against the process's current working
directory — not the file where the code lives. For most use cases these are the same thing.
You run your script from its directory, `"arith.wasm"` is right next to it, done.

But if you open a Jupyter notebook in `/project/` and import from `/project/src/`:

```
project/
  notebook.ipynb         ← CWD when running
  src/
    quantum_ops.py       ← contains @wasm_module("arith.wasm")
    arith.wasm           ← lives next to quantum_ops.py
```

The path `"arith.wasm"` resolves to `/project/arith.wasm` — which doesn't exist. The
file the decorator is written in knows exactly where it is, but nobody asked it.

## Why the existing helper wasn't enough

Guppy already had a `get_calling_frame()` helper in `decorator.py`. Its job was to walk
the Python call stack and skip frames that belong to guppylang internals, returning the
first frame from user code. That would give you `__file__` on the user's module.

The problem: `@wasm_module` is wrapped by `@pretty_errors`. So the call stack when
`@wasm_module` runs looks like this:

```
Frame 0: decorator.py         ← guppylang internal
Frame 1: decorator.py         ← @pretty_errors wrapper (also guppylang!)
Frame 2: quantum_ops.py       ← user's file (what we want)
Frame 3: notebook.ipynb       ← jupyter kernel (not what we want)
```

The old `get_calling_frame()` only skipped frames from `decorator.py` specifically.
But `@pretty_errors` added another guppylang frame, so the skip logic stopped one
frame too early and returned the wrong location.

<Callout type="note" label="What @pretty_errors does">
`@pretty_errors` is a decorator applied to decorator factories in Guppy. It wraps the
decorated function so that any `GuppyError` thrown during decoration gets formatted
with a nice traceback. It's useful, but it means every decorator call has an extra
guppylang frame on the stack.
</Callout>

## The fix: skip all guppylang frames

The fix generalizes the frame-skipping logic. Instead of checking for one specific file,
it skips any frame whose file lives inside the guppylang package directory:

```python
def get_calling_frame() -> FrameType | None:
    """Walk the call stack past all guppylang-internal frames."""
    guppylang_dir = str(Path(__file__).parent.parent)  # package root

    frame = sys._getframe(1)
    while frame is not None:
        filename = frame.f_code.co_filename
        if not filename.startswith(guppylang_dir):
            return frame
        frame = frame.f_back
    return None
```

Any frame whose filename is inside the guppylang package directory gets skipped. The
first frame outside that directory is the user's code — no matter how many layers of
internal wrappers are between them.

## Resolving the path

With the calling frame in hand, the resolution is straightforward:

```python
def resolve_wasm_path(path: str | Path) -> Path:
    resolved = Path(path)
    if resolved.is_absolute():
        return resolved

    # Try to resolve relative to the calling file
    frame = get_calling_frame()
    if frame is not None and (caller_file := frame.f_code.co_filename):
        caller_dir = Path(caller_file).parent
        return (caller_dir / resolved).resolve()

    # Fall back to CWD (interactive sessions, eval'd code, etc.)
    return resolved.resolve()
```

Absolute paths pass through unchanged — this only changes behavior for relative paths.
The fallback to CWD handles interactive Python sessions and `eval()`-based contexts
where there's no backing file.

<Callout type="warning" label="The fallback case">
If `get_calling_frame()` returns `None` — which can happen in a REPL or inside
`exec()` — the code falls back to CWD resolution. This is the same behavior as before
the fix, so nothing regresses for those contexts. The fix is additive.
</Callout>

## The four test cases

The PR adds four scenarios to the test suite, which neatly cover the problem space:

**1. Relative path, different working directory** — the core fix. A test that changes
`os.chdir()` to a different directory before decorating, then checks the WASM file is
still found next to the source file.

**2. Absolute path, different working directory** — regression prevention. Absolute
paths should be unaffected by CWD. Verifying this ensures the fix didn't accidentally
break something in the happy path.

**3. Absolute path with CWD changes** — robustness. Confirms absolute paths work even
if CWD has been changed multiple times before decoration runs.

**4. Nonexistent relative path** — error handling. Even after the fix, passing a path
that doesn't exist should still raise a clear error, not silently succeed.

The fourth case is easy to overlook. After fixing path resolution, it would be easy
to accidentally make bad paths fail silently if the lookup logic returned `None` and
the code swallowed it. The explicit test catches that.

## What I found interesting

The bug required understanding two separate Python mechanisms: how relative paths resolve
(OS-level CWD), and how Python's call stack is structured (frame inspection). Neither
is something you think about often.

The `@pretty_errors` wrapping is what made the existing frame-skip logic insufficient.
It's a reminder that decorators compose invisibly — every layer added to a decorator
factory adds a frame to the stack, and any code that inspects frames needs to account
for all of them.

The fix is robust to future additions too. By checking against the package directory
rather than a specific filename, any new internal wrappers added to guppylang won't
break frame discovery again.
