`comptime()` is one of Guppy's more powerful features — it lets you run arbitrary Python
at compile time to compute values that get embedded in the compiled program. But it has a
sharp edge: you can't use Guppy constructs inside it. If you try, you used to get a raw
Python traceback that gave no useful hint about what went wrong.

PR #1527 fixes this with a proper diagnostic.

## What `comptime()` is

In Guppy, `comptime()` is an escape hatch for compile-time computation:

```python
@guppy
def make_rotation(module: GuppyModule) -> qubit:
    angle = comptime(math.pi / 4)   # evaluated at compile time
    return rx(angle)
```

The expression inside `comptime()` runs in a normal Python environment. It can call Python
functions, do math, access Python objects — anything that's valid Python. The result gets
lifted into the Guppy program as a constant.

The critical constraint: you can't call Guppy types or Guppy-decorated functions inside
`comptime()`. Guppy functions don't exist as callable Python objects at compile time — they're
being *compiled*, not executed.

## The mistake and the old error

If you accidentally use a Guppy type or function inside `comptime()`:

```python
@guppy
def bad_comptime() -> int:
    return comptime(int(42))   # int is a Guppy type here, not Python's int
```

Before PR #1527, this would produce something like:

```
Traceback (most recent call last):
  File ".../guppylang/checker/expr_checker.py", line 312, in _eval_comptime
    result = eval(compiled, globals_dict)
  File "<comptime>", line 1, in <module>
TypeError: 'GuppyType' object is not callable
```

That traceback is coming from inside Guppy's evaluation machinery. The user sees
`GuppyType object is not callable` with a stack frame pointing into the checker internals
— not helpful at all.

<Callout type="warning" label="The @hide_trace complication">
There was a secondary bug: the `@hide_trace` decorator (used elsewhere in Guppy to
suppress internal tracebacks) was overwriting `sys.excepthook`. This meant that even
errors that *were* formatted as proper GuppyErrors would sometimes display as raw Python
tracebacks. PR #1527 fixes this too with a `saved_exception_hook()` context manager.
</Callout>

## The fix: catching `GuppyComptimeError`

The solution is in `expr_checker.py`, in the `_eval_comptime()` method. The evaluation
now runs inside a try/except that specifically catches errors from Guppy objects being
invoked in a comptime context:

```python
def _eval_comptime(self, expr: ast.expr) -> Any:
    with saved_exception_hook():
        try:
            result = eval(compiled_expr, self._comptime_globals)
        except GuppyComptimeError as e:
            # A Guppy construct was called inside comptime() — format a proper error
            raise GuppyError(ComptimeGuppyObjectError(expr, e.guppy_object)) from None
    return result
```

`GuppyComptimeError` is raised by Guppy types and functions when they're called from a
Python context (rather than being compiled). The `ComptimeGuppyObjectError` diagnostic
then formats a user-facing message.

## The new `ComptimeGuppyObjectError` diagnostic

The new diagnostic class in `checker/errors/comptime_errors.py` receives the offending
expression and the Guppy object that was called:

```python
@dataclass
class ComptimeGuppyObjectError:
    expr: ast.expr         # The comptime() call site
    obj: GuppyDefinition   # The Guppy type or function that was invoked
```

This produces an error like:

```
GuppyError: Cannot use Guppy type `int` inside `comptime()`
  --> my_file.py:3
  |
3 |     return comptime(int(42))
  |                     ^^^
  |
help: `int` is a Guppy type. To use the Python built-in `int`, import it
      explicitly or use it outside of `comptime()`.
```

The `help:` line is the important addition — it explains *why* this fails and what the
user should do instead. Often the mistake is that the user intends to use Python's `int`
but the name has been shadowed by the Guppy type.

## The `saved_exception_hook()` context manager

The secondary fix is worth understanding. The `@hide_trace` decorator sets `sys.excepthook`
to suppress Python tracebacks. But once it's set, it stays set — which could swallow
errors that should be shown as proper Guppy diagnostics.

```python
from contextlib import contextmanager

@contextmanager
def saved_exception_hook():
    """Temporarily preserve the original sys.excepthook."""
    original = sys.excepthook
    try:
        yield
    finally:
        sys.excepthook = original
```

This context manager saves and restores `sys.excepthook` around comptime evaluation.
Any modifications made during evaluation — including by `@hide_trace` on called functions —
are undone when the context exits. Errors flow through the proper Guppy error pipeline.

## What I found interesting

The bug structure here is two things compounding each other. The first is a missing error
handler (no catch for `GuppyComptimeError`). The second is a side effect persisting longer
than intended (`sys.excepthook` being permanently overwritten). Each problem on its own
would be annoying; together they made a confusing class of mistakes completely opaque.

The fix is also clean: one new context manager, one new `except` clause, one new diagnostic
class. The `saved_exception_hook()` pattern is immediately reusable anywhere Guppy does
similar dynamic evaluation.
