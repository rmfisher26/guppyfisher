There's a class of bug that feels like it should obviously be caught at compile time, but
wasn't. You pass `qubits[0]` to a function, and then you pass `qubits[0]` again in the
same call. Both arguments look valid. The compiler says nothing. Then at runtime: a panic.

PR #1469 fixes this. It adds compile-time detection for duplicate literal array subscripts
in function call arguments. This post traces exactly how.

## Why this is a linearity violation

If you have an array of qubits and you index into it with the same literal index twice in
the same call, you're trying to use the same qubit in two places simultaneously. That's a
linearity violation — the no-cloning theorem prevents it.

```python
@guppy
def bad(qubits: array[qubit, 4]) -> None:
    cx(qubits[0], qubits[0])  # Both args are the same qubit
```

The type system can only prevent this if it understands that `qubits[0]` and `qubits[0]`
refer to the *same* value. For variables, the linearity checker already handles this — it
tracks which names have been consumed. But array subscripts with literal indices are a
different shape of expression.

<Callout type="note" label="Why runtime?">
Before this PR, the compiler would generate HUGR nodes for both uses of `qubits[0]`. The
runtime would then encounter an invalid graph where the same wire had two consumers — and
panic. The bug existed silently in the IR.
</Callout>

## What the PR adds: two helper functions

The core of the fix lives in `linearity_checker.py`, in the `_visit_call_args()` method.
Two new helpers do the work:

```python
def get_literal_index(node: ast.expr) -> int | None:
    """If this expression is a simple integer literal, return its value."""
    if isinstance(node, ast.Constant) and isinstance(node.value, int):
        return node.value
    return None

def is_simple_literal_subscript(node: ast.expr) -> tuple[str, int] | None:
    """If this is arr[n] where n is a literal int, return (arr_name, n)."""
    if (
        isinstance(node, ast.Subscript)
        and isinstance(node.value, ast.Name)
    ):
        idx = get_literal_index(node.slice)
        if idx is not None:
            return (node.value.id, idx)
    return None
```

Together these identify expressions of the form `name[literal]`. Not `arr[i]` where `i`
is a variable — just the cases where the index is a known integer constant.

## Tracking duplicates inside `_visit_call_args()`

With these helpers in place, `_visit_call_args()` builds a set of seen subscripts as it
processes each argument:

```python
seen_subscripts: set[tuple[str, int]] = set()

for arg in call_args:
    subscript = is_simple_literal_subscript(arg)
    if subscript is not None:
        arr_name, idx = subscript
        # Look up the type of the array element
        ty = self._get_type(arg)
        if subscript in seen_subscripts and not is_copyable(ty):
            raise AlreadyUsedError(arg, arr_name, idx)
        seen_subscripts.add(subscript)
```

If the same `(name, index)` pair appears twice and the element type is non-copyable — which
qubits are — it raises `AlreadyUsedError` immediately, pointing at the second use.

<Callout type="warning" label="Scope">
This only catches literal indices. `cx(qubits[i], qubits[i])` where `i` is a variable
is still not caught at compile time — that requires more sophisticated alias analysis.
PR #1469 is explicit about this being a "best effort" improvement for the common case.
</Callout>

## What the error looks like now

Before PR #1469:

```
RuntimeError: HUGR validation failed: node has no input wire
```

After:

```
GuppyError: Qubit `qubits[0]` was already used here
  --> my_file.py:3
  |
3 |     cx(qubits[0], qubits[0])
  |                   ^^^^^^^^^
  |
note: `qubits[0]` was first used here
  |
3 |     cx(qubits[0], qubits[0])
  |        ^^^^^^^^^
```

The difference in debugging experience is significant. A runtime panic in HUGR validation
gives you no indication of which Python-level expression caused the problem. The
compile-time error tells you exactly which line and which argument.

## The tests

Three new error test files cover the cases:

- `duplicate_subscript_literal.py` — owned qubit, used twice as a move
- `duplicate_subscript_owned.py` — variant with non-qubit linear type
- `duplicate_subscript_borrow.py` — same index borrowed twice

The borrow case is worth noting: even if you're *borrowing* `qubits[0]`, you can't borrow
the same element twice in the same call. The aliased borrow is still a violation.

## What I took away

This PR is a good example of the linearity checker being incrementally strengthened. The
checker can't yet handle all aliasing through array indices — but it can now handle the
simplest and most common case. Each improvement like this narrows the gap between "compiles"
and "correct".

The two helper functions are small and easy to reason about. That's by design — a more
general solution would require tracking arbitrary alias relationships, which is a much
bigger piece of work.
